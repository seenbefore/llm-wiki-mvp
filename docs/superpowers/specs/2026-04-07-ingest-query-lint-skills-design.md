# Ingest Query Lint Skills Design

## 目标

把当前仓库中已经文档化的 `ingest`、`query`、`lint` 工作流，提升为 3 个可稳定调用的仓库本地 skill：

- `ingest`
- `query`
- `lint`

这些 skill 直接作为调用入口存在，目标是让用户和 Agent 在本仓库内能够用接近“指令”的方式触发操作，同时把当前仓库已经建立的页面模型、Obsidian 链接规范、索引与日志要求稳定地固化下来。

## 背景

当前仓库已经有较完整的知识库底座：

- `schema/AGENTS.md`：通用行为约束
- `schema/workflows.md`：人类可读的 `ingest/query/lint` 流程说明
- `schema/templates/`：来源页、主题页、概念页、实体页、分析页模板
- `wiki/index.md`：内容索引
- `logs/log.md`：时间线记录

目前的问题不是“没有规则”，而是“缺少稳定入口”。这些能力还更像文档约定，而不是可重复调用的仓库操作。

## 设计决策

- 不做一个总控 skill。
- 不再额外包一层命令路由。
- 直接在仓库内提供 3 个本地 skill：`ingest`、`query`、`lint`。
- skill 是执行入口，`schema/` 是规则底座，二者职责分离。
- 这 3 个 skill 先只服务当前仓库，不追求一开始抽象成通用知识库框架。

## 总体架构

### 角色分工

- `ingest`：把 `raw/` 中的新资料编译进 `wiki/`
- `query`：基于现有 `wiki/` 回答问题，并在需要时沉淀分析页
- `lint`：巡检并修复知识库结构与内容健康问题

### 依赖关系

- skill 读取 `schema/AGENTS.md`，继承仓库总规则
- skill 读取 `schema/workflows.md`，继承仓库流程语义
- skill 读取 `schema/templates/`，生成或更新具体页面
- skill 更新 `wiki/index.md`
- skill 追加 `logs/log.md`

这意味着 skill 本身不重新定义页面模型，而是负责把现有模型“执行化”。

## Skill 1: ingest

### 目的

将一份或多份 `raw/` 资料转化为结构化知识页，并将结果纳入索引和日志。

### 最小输入

- 一个来源路径，或一组来源路径

推荐输入形式：

- 单文件 ingest：`raw/articles/foo.md`
- 批量 ingest：`raw/articles/` 或一个显式文件列表

### 主要输出

- 至少 1 个 `wiki/sources/` 来源页
- 0 到多个 `wiki/topics/`、`wiki/concepts/`、`wiki/entities/` 的增量更新
- `wiki/index.md` 更新
- `logs/log.md` 追加一条 ingest 记录

### 运行约束

- 不得修改 `raw/` 原始资料内容
- 优先做增量更新，不整页重写已有知识页
- 对来源不足、冲突未消解或置信度不够的内容，只落来源页，不强行写综合结论
- 新增或更新链接必须遵守当前仓库的 Obsidian 链接规范

### 成功标准

- 新资料被纳入 `wiki/`
- 结果可从 `wiki/index.md` 导航到
- `logs/log.md` 中可追踪本次 ingest
- 不引入新的空白双链目标

## Skill 2: query

### 目的

基于当前 `wiki/` 中已经沉淀的内容回答问题，而不是绕回原始资料做一次性回答。

### 最小输入

- 一个问题

例如：

- “LLM Wiki 和传统 RAG 的核心区别是什么？”
- “当前仓库里 Obsidian 链接规范的推荐写法是什么？”

### 主要输出

- 一次基于已有 wiki 的回答
- 在回答具备长期复用价值时，新增或更新 `wiki/analyses/` 页面
- 如分析页形成了新结构或明确结论，可回写主题页或概念页
- `logs/log.md` 追加一条 query 沉淀记录

### 运行约束

- 默认先读 `wiki/index.md`
- 优先引用现有 `wiki/` 页面而不是直接引用 `raw/`
- 除非用户明确要求，否则不把“重新读原始资料”作为主回答路径
- 若问题超出当前 wiki 覆盖范围，应明确说明信息缺口，而不是假装已有结论

### 成功标准

- 回答可追溯到已有 wiki 页面
- 高价值回答能够沉淀，而不是只停留在会话里
- 若回写分析页，则索引和日志同步更新

## Skill 3: lint

### 目的

定期检查知识库结构与内容健康度，并在低风险范围内进行修正。

### 最小输入

- 可为空

默认行为：

- 无输入时巡检全库
- 也允许局部 lint，例如只检查 `wiki/concepts/` 或某一批近期更新页面

### 主要输出

- 一个问题清单
- 对低风险问题的直接修复，或对高风险问题的明确报告
- `logs/log.md` 追加一条 lint 记录

### 检查范围

- 孤儿页
- 无来源结论
- 重复概念或重复实体
- 索引遗漏
- 链接失效
- 未标记的冲突
- 不符合仓库 Obsidian 链接规范的双链写法

### 风险分级

- 低风险：索引漏链、明显的错链、格式级 frontmatter 问题、日志漏记
- 中风险：分析页与主题页之间的结构回写
- 高风险：重写综合结论、合并两个概念页、删除页面

### 成功标准

- 低风险问题被稳定修复
- 高风险问题被清楚报告，不被静默处理
- 巡检结果可在日志中追踪

## 仓库结构建议

建议把 3 个 skill 放到仓库本地、易发现的位置，并保持平级：

```text
skills/
  ingest/
    SKILL.md
  query/
    SKILL.md
  lint/
    SKILL.md
```

说明：

- 每个 skill 的 `SKILL.md` 只定义自身职责、输入、输出、依赖规则和失败处理
- 不复制整套 schema 内容，只引用当前仓库的规则文件
- 这样未来修改模板或链接规范时，不需要同时维护两份真相来源
- 实际实现目录为 `skills/ingest/`、`skills/query/`、`skills/lint/`

## 与现有 schema 的关系

### `schema/AGENTS.md`

继续作为仓库总规则，定义全局行为边界，例如：

- 不改 `raw/`
- 增量更新优先
- 链接规范
- 低置信度处理

### `schema/workflows.md`

继续作为人类可读的流程说明，适合作为 skill 的语义底座。

### `schema/templates/`

继续作为知识页模板来源。skill 生成新页面时优先参考这里，而不是在 skill 内硬编码页面格式。

## Obsidian 链接规范要求

这 3 个 skill 都必须遵守当前仓库已经形成的约定：

- 文件名优先用 `kebab-case`
- 正文优先使用 `[[文件名|显示文本]]`
- 只有在真实文件名或 `aliases` 精确匹配时，才允许直接写 `[[目标名]]`
- `aliases` 必须是 YAML 列表

`lint` 需要把“链接规范巡检”纳入默认检查项，`ingest` 和 `query` 需要把这条规则视为写入约束。

## 失败处理

### ingest

- 资料过薄：只生成来源页
- 来源冲突：标 `冲突/待核实`，不抢跑综合结论
- 页面归类不确定：优先写来源页并把待跟进写清楚

### query

- wiki 覆盖不足：明确指出信息缺口
- 回答不值得沉淀：只回答，不强行生成分析页

### lint

- 能低风险修复的：直接修
- 需要内容判断的：报告并等待人工确认

## 验证策略

实现后，每个 skill 至少应验证以下事项：

### ingest

- 是否生成来源页
- 是否更新索引
- 是否追加日志
- 是否引入错误双链

### query

- 是否先从 `wiki/index.md` 和相关页面开始
- 是否优先引用已有 wiki 页面
- 若落分析页，是否同步索引和日志

### lint

- 是否能检出索引遗漏、错链、frontmatter 问题
- 是否区分低风险自动修复与高风险人工确认

## 非目标

当前设计不包括：

- 通用化到其他知识库仓库
- 自动联网检索作为 skill 默认能力
- 完整 CLI、脚本系统或 GUI
- 大规模标题块链接与块级引用规范

## 推荐实施顺序

1. 先写 `ingest` skill，因为它最直接承接已有资料流
2. 再写 `query` skill，因为它要依赖已沉淀的 wiki 结构
3. 最后写 `lint` skill，因为它需要同时理解前两者写出来的结果

## 预期结果

完成后，这个仓库会从“有一套写得不错的工作流文档”，升级为“有 3 个稳定本地 skill 的可执行知识库系统”。你不再需要每次重新描述怎么 ingest、怎么 query、怎么 lint，仓库本身会把这些操作方式固化下来。
