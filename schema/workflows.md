# Workflows

这个文件描述 MVP 阶段的五条核心工作流：`task-start`、`task-close`、`ingest`、`query`、`lint`。

## Task Start

适用场景：用户准备开始一个超过 30 分钟的重要 AI 任务，需要先复用当前 Wiki 中的已有上下文。

步骤：

1. 读取 `schema/AGENTS.md` 与 `wiki/index.md`。
2. 按任务描述定位相关主题页、概念页、实体页、分析页和来源页。
3. 汇总已有上下文、可复用结论和信息缺口。
4. 输出本次任务建议加载的上下文包。
5. 暂停在任务启动结果，不直接执行任务。

最小产出：

- 1 份任务启动上下文
- 0 个文件变更

## Task Close

适用场景：一个重要任务结束后，需要判断本次结果是否值得沉淀，并为后续 ingest 准备原始记录。

步骤：

1. 回顾任务目标、过程、结论、命令、踩坑和开放问题。
2. 判断是否值得沉淀到 LLM Wiki。
3. 若值得，默认写入 `raw/task-notes/YYYY-MM-DD-<task-slug>.md`。
4. 不直接修改 `wiki/`；需要进入知识层时，后续再对该 raw 记录执行 ingest。
5. 输出建议的下一步 ingest 命令和 git commit message。

最小产出：

- 0 到 1 个 raw task note
- 1 条下一步 ingest 建议

## Ingest

适用场景：`raw/` 中新增了一篇文章、一个 PDF、一份笔记或一批附件。

步骤：

1. 读取原始资料并确认它属于什么来源类型。
2. 在 `wiki/sources/` 新建一个来源摘要页。
3. 提炼关键事实、主要观点、可复用结论和待核实点。
4. 判断它影响哪些 `topics/entities/concepts` 页面。
5. 对相关知识页做增量更新，而不是整页重写。
6. 更新 `wiki/index.md`。
7. 在 `logs/log.md` 记录本次 ingest。

最小产出：

- 1 个来源页
- 0 到多个知识页增量更新
- 1 条日志记录

## Query

适用场景：用户基于已有 Wiki 提问，希望得到可追溯答案。

步骤：

1. 先看 `wiki/index.md`，定位相关页面。
2. 读取相关主题页、概念页、实体页和来源页。
3. 产出回答时优先引用现有 Wiki 页面。
4. 若回答具备长期复用价值，在 `wiki/analyses/` 落一页。
5. 若分析页引入了新结构或结论，回写相关主题页并更新索引。
6. 在 `logs/log.md` 记录这次沉淀。

## Lint

适用场景：定期健康检查，或在摄入了一批资料后做整理。

步骤：

1. 检查是否有孤儿页。
2. 检查页面是否缺少来源引用。
3. 检查是否有重复概念或重复实体页。
4. 检查链接是否失效或索引未更新。
5. 检查是否有旧结论已被新来源挑战但尚未标记。
6. 把发现的问题记到 `logs/log.md`，必要时修复页面。

建议频率：

- 小规模日常使用：每周一次
- 连续 ingest 多份资料后：立即做一次
